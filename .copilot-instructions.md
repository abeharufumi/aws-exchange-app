# aws-exchange-app 開発方針（AI 常設指針）

## 基本ルール

- **返答言語**: 日本語。簡潔・非人格的。
- **作業方針**: 質問せず即実行。不明点は調査後に実装。

## プロジェクト構造

### フロントエンド構造（`frontend/`）

- **起動**: `cd frontend && npm run web`
- **ルーティング**: Expo Router 統一
- **フォルダ構成**:
- `app/`: 画面＆ルーティング定義（URL に対応）
  - `(tabs)/index.tsx` → Timeline（Home）
  - `(tabs)/explore.tsx` → Search
  - `(tabs)/matches.tsx` → Matches
  - `(tabs)/notification.tsx` → Notification
  - `(tabs)/profile.tsx` → Profile
  - `chat/[id].tsx` → Chat詳細（動的ルート）
  - `user/[id].tsx` → 他ユーザープロフィール詳細（動的ルート）
  - `auth/index.tsx` / `auth/login.tsx` / `auth/signup.tsx` → 認証ルート
  - `qr/[meetRequestId].tsx` → QR詳細（動的ルート）
  - `review/[meetId].tsx` → Review詳細（動的ルート）
  - `boost-purchase.tsx` → Boost購入画面
  - `premium-subscribe.tsx` → Premium加入画面
  - `footprint.tsx` → 足跡履歴画面
  - `live.tsx` → ライブ配信一覧画面
  - `live/[streamId].tsx` → ライブ配信詳細画面
  - `fanclub.tsx` → 自分のファンクラブ管理画面
  - `fanclub/[creatorId].tsx` → クリエーターのファンクラブ詳細画面
  - `call-tickets.tsx` → 通話チケット画面
  - `gifts.tsx` → ギフト画面
  - `_layout.tsx` → レイアウト定義
- ルート直下の補助ファイル:
  - `app.json`: Expo 設定
  - `package.json`: 依存管理とスクリプト
  - `tsconfig.json`: TypeScript 設定
  - `babel.config.js`: Babel 設定
  - `tailwind.config.js`: 互換維持用（NativeWind未使用）
  - `nativewind-env.d.ts`: 互換維持用（参照定義は空）
  - `global.css`: 互換維持用（Tailwindディレクティブは未使用）
  - `expo-env.d.ts`: Expo 環境型定義
  - `README.md`: フロントエンド用ドキュメント
- `assets/`: 画像・フォントなど静的アセット
  - `images/`: 画像フォルダ
- `components/`: アプリ共通UI（テンプレ由来の共通部品）
  - `ui/`: 小さな共通 UI コンポーネント群（`collapsible.tsx` 等）
- `constants/`: テーマ・定数（例: theme.ts）
  - `apiConstants.ts`: APIベースURL/パス定数
- `contexts/`: 互換用の再エクスポート（新規追加禁止。`src/screens/` が `../../contexts/auth` でimportする互換層として保持）
  - `auth.tsx`: `src/contexts/auth.tsx` への再エクスポートのみ
- `hooks/`: 共通フック（テンプレ由来 + 追加分）
- `lib/`: ライブラリ初期化用（現在は `prisma.ts` のみ。`api-client.ts` は削除済み・API通信は `src/services/api.ts` に統一）
  - `prisma.ts`: Prisma クライアント初期化（補助）
- `scripts/`: 開発補助スクリプト（例: reset-project.js）
  - `reset-project.js`: プロジェクト初期化スクリプト
- `prisma/`: Prisma スキーマ（ツール連携用）
  - `schema.prisma`
  - `prisma.config.ts` は `frontend/` 直下に配置
- `src/`: ロジック＆コンポーネント
  - `contexts/`: 認証などアプリ全体の状態管理（AuthProvider 等）
  - `screens/`: 画面コンポーネント（app/ から import される）
    - `matches/`: マッチ一覧画面
    - `boost/`: Boost購入画面
    - `premium/`: Premium加入画面
    - `footprint/`: 足跡履歴画面
    - `user/`: 他ユーザープロフィール詳細画面
    - `live/`: ライブ配信画面
    - `fanclub/`: ファンクラブ画面
    - `call-ticket/`: 通話チケット画面
    - `gift/`: ギフト画面
  - `components/`: 再利用可能な UI コンポーネント
    - `common/`: 小さく責務を絞った共通 UI
      - `LoadingState.tsx`
      - `EmptyState.tsx`
      - `ScreenBackButton.tsx`
      - `SectionCard.tsx`
      - `ActionButton.tsx`
      - `ActionButtonRow.tsx`
      - `ReceiveFilterToggle.tsx`
      - `StatusInfoBox.tsx`
      - `SegmentedTab.tsx`
  - `constants/`: フロントエンド内部定数
    - `apiConstants.ts`: APIベースURL/パス定数
  - `services/`: API 呼び出し（api.ts など）
  - `theme/`: 画面別テーマスタイル定義（`src/theme/*`）
    - `index.ts`
    - `createThemeStyles.ts`
    - `customTheme.ts`（react-native-paperのアプリ共通テーマ）
    - `Home.ts` / `Search.ts` / `Matches.ts` / `UserDetail.ts` など
  - `hooks/`: カスタムフック
  - `types/`: TypeScript 型定義
    - `auth.ts`
    - `chat.ts`
    - `user.ts`
    - `profile.ts`
    - `notification.ts`
    - `match.ts`
    - `meet.ts`
    - `live.ts`
    - `gift.ts`
    - `fanclub.ts`
    - `call-ticket.ts`
    - `boost.ts`
    - `premium.ts`
    - `icon-frame.ts`
  - `utils/`: ユーティリティ関数
    - `authStorage.ts`: 認証情報の保存/復元/削除（token key移行を含む）
    - `rankProgress.ts`: ランク進捗表示用の共通ロジック
  - 注記: `src/stores/` は廃止。認証は `src/contexts/auth.tsx` に一本化
- `package.json`: main: "expo-router/entry" 固定
  - 注記（重要）: `app/` は基本的にルーティング用エントリ（軽いラッパー）に限定し、**画面本体の主要実装は `src/screens/` に置く**こと。
    - `app/` 内のファイルは `src/screens/*` から該当コンポーネントを import して返すラッパーにするのが標準運用。
    - 例外（既存で直接実装を含むファイル）: `app/_layout.tsx`（ルートレイアウト）、`app/(tabs)/_layout.tsx`（タブレイアウト）、`app/modal.tsx`（モーダル）。
    - 認証ルートは `app/auth/index.tsx` / `app/auth/login.tsx` / `app/auth/signup.tsx` をルーティング用ラッパーとして運用し、実装本体は `src/screens/auth/*` に置く。
    - 新規に画面を追加する場合はまず `src/screens/` に実装を作り、`app/` 側はルーティング用の薄いラッパーを作成してください。

### バックエンド構造（`backend/`）

- **起動**: `cd backend && source .venv/bin/activate && uvicorn main:app --reload`
- **データベース**: PostgreSQL (Docker Compose)
- **フォルダ構成**:
  - `main.py`: エントリーポイント（ルーター統合のみ）
  - `routers/`: エンドポイント定義（役割ごとに分割）
    - `auth.py` → `/auth/*` (ログイン・サインアップ)
    - `users.py` → `/api/users/*` (ユーザー検索・プロフィール)
    - `matches.py` → `/api/matches/*` (Like・Pass)
    - `notifications.py` → `/api/notifications/*` (通知)
    - `chat.py` → `/api/chat/*` (チャット)
    - `meets.py` → `/api/meet/*` (デート予約・QR)
    - `reviews.py` → `/api/review/*` (レビュー)
    - `boost.py` → `/api/boost/*` (ブースト購入・有効化)
    - `premium.py` → `/api/premium/*` (プレミアム加入・解約)
  - `utils/`: 共通ユーティリティ
    - `security.py` → JWT・パスワードハッシュ
    - `dependencies.py` → 認証・依存性注入（`get_current_user` のみ。`Authorization: Bearer` 必須、`get_current_user_or_first_active` は削除済み）
    - `rank.py` → ランク昇格・降格共通ロジック（`meets.py` / `reviews.py` から呼び出し）
  - `database.py`: DB 接続・セッション管理
  - `execQuery.py`: 共通クエリ実行関数（execute_select, execute_insert など）
  - `schema.sql`: データベーススキーマ定義（CREATE TABLE文）
  - `seed.sql`: テストデータ投入用SQLスクリプト（開発用）
  - `scripts/`: AWS運用補助スクリプト
    - `init_db.py`: `schema.sql` / `seed.sql` をRDSへ適用
    - `deploy_ecs.py`: ECSサービスを最新イメージで更新
    - `run_init_task.py`: FargateワンショットでDB初期化実行
    - `fix_cloudfront_alb_origin_protocol.py`: CloudFront ALBオリジン接続方式の修正
    - `update_cloudfront_auth_behavior.py`: CloudFront `/auth/*` ビヘイビア追加
    - `remove_cloudfront_auth_behavior.py`: CloudFront `/auth/*` ビヘイビア削除
  - `.venv/`: Python 仮想環境

### データベーススキーマ（`schema.sql`）

- **用途**: テーブル作成（初回セットアップ時）
- **実行方法**:
  ```bash
  psql -U postgres -d aws_exchange_app -f backend/schema.sql
  ```
- **内容**: 全31テーブル（P1: 11個、P2: 8個、P3: 12個）
- **注意**: ORM（models.py）は使用せず、純粋なSQL方式で統一
- **🔴重要**: `schema.sql` を更新した際は、必ず `schema-erd.md` のMermaidER図も同時更新すること

### テストデータ投入（`seed.sql`）

- **用途**: 開発・テスト用のサンプルデータ投入
- **実行方法**:

  ```bash
  # Dockerコンテナ内で実行
  docker-compose exec db psql -U postgres -d aws_exchange_app -f /app/seed.sql

  # またはローカルから
  psql -U postgres -d aws_exchange_app -f backend/seed.sql
  ```

- **内容**: ユーザー28件（Rank5/4/3/2/1/0各層）、チャット14件、デート5件、通知18件、マッチング22件
- **注意**: ORM（`db.add()`など）ではなく純粋なSQL方式で統一
  - 補助ファイル/スクリプト（実在するもの）:
    - ルート: `docker-compose.yml`, `mermaid.md`, `feMaleUserPrivilegeList.md`, `maleUserPrivilegeList.md`
    - `backend/`: `__pycache__/`（実行時生成）
    - `frontend/`: `prisma/`（`schema.prisma`）、`prisma.config.ts`、`app.json`、`expo-env.d.ts`, `tsconfig.json`

## コード生成の制約

**🔴重要: ファイル作成やコード編集前に必ずこのファイルを参照してください。**

**🔴追加指示: 変更を加えるたびにこのファイルの「- フォルダ構成:」を最新化すること**

**フォルダ階層を絶対遵守**:

- Frontend のロジックは `src/` に、画面定義は `app/` に配置
- Backend のエンドポイントは `routers/` に、共通ロジックは `utils/` または `execQuery.py` に配置
- 新規UIは `StyleSheet.create` を増やさない
- **🔴UIスタイリングは `react-native-paper` + インライン `style={{ }}` のみで統一する**
  - NativeWind / tailwindcss は**完全除去済み・再導入禁止**
  - `className=` は**絶対に使用禁止**
  - スタイルは `react-native-paper` コンポーネントの props（`mode`, `buttonColor`, `style`, `contentStyle` 等）+ インライン `style={{ ... }}` のみ
  - レイアウト・余白・色はインライン style オブジェクトで直接指定する
  - `StyleSheet.create` の追加は原則禁止。共通テーマは `src/theme/customTheme.ts` に集約する
- **リポジトリパターン・サービスクラスは使わない**（SQL 直書き + execQuery 方式）
- 複数の役割が混在するファイル（例: `main.py` に全エンドポイント）は作らない
- 新規ファイル作成時は必ず適切なフォルダに配置し、命名規則に従う

**共通化の実装スタンス**:

- 同じ責務・同じ見た目・同じ操作が複数画面で繰り返される場合は、`src/components/` に共通コンポーネントとして切り出す
- ただし、**先回りした過度な抽象化は禁止**。利用箇所が1画面のみ、または props が増えすぎて責務が曖昧になる場合は共通化しない
- 共通化は **UIの重複削減と可読性向上が明確な場合に限定** し、ドメイン固有ロジックまで無理にまとめない
- まずは `Header`、`Loading`、`EmptyState`、`ButtonRow` など、責務が小さく再利用性が高い単位から共通化する
- 団長なコード（密結合・巨大コンポーネント・多責務コンポーネント）を避け、1コンポーネント1責務を徹底する
- カード、リスト行、モーダル枠などの**定型レイアウトは中身を `children` で受け取り、構造（外枠・余白・境界線）だけを共通化**する
- UIを伴わない状態管理・バリデーション・変換などの**純粋ロジックは `src/hooks/` または `src/utils/` に切り出して共通化**する
- 上記ロジック共通化でも、画面固有の文言・遷移・API呼び出しまでまとめない（責務分離を維持）

**Python 型注釈の統一**:

- オプショナル型は `Optional[str]` を使用（`str | None` は使わない）
- 例: `avatarUrl: Optional[str] = None`
- `from typing import Optional` をインポート

**TypeScript/JavaScript 型注釈の統一**:

- オプショナル型は `?` を使用
- 例: `avatarUrl?: string`

**🔴エンドポイントクラス定義の必須ルール**:

- **すべての Response/Request クラスには日本語コメントでパラメーター説明を必須記載**
- 特に選択肢がある場合（status, type など）は可能な値を列挙すること
- 例:
  ```python
  class MeetRequestResponse(BaseModel):
      id: int
      status: str  # 'pending', 'accepted', 'rejected', 'completed'
      scheduled_date: str
      scheduled_time: str
  ```

## コード提示ルール

- Markdown コードブロック: 4 バッククォート開始/終了
- 1 行目に `// filepath: /absolute/path/to/file.ext` 必須
- 既存コード省略: `// ...existing code...`
- **スタイル管理**:
  - MUIライクな方針を採用する（デフォルト継承 → `customTheme` で最小差分 → 画面固有は `props` 上書き）
  - React Native向けOSS UIコンポーネントライブラリは `react-native-paper` を標準採用する
  - 新規スタイルはまずライブラリコンポーネントで解決し、`StyleSheet.create` の追加は原則禁止
  - `className=` は使用禁止。スタイルはインライン `style={{ }}` のみ
  - 既存のテーマ共通化は `src/theme/` へ配置し、`src/theme/index.ts` から参照する
  - 使用例: `<View style={{ flex: 1, backgroundColor: '#fff' }} />`
- 複数ファイル編集: multi_replace_string_in_file 使用
